# Інструкція з типізації в проєкті

## Мета

Цей документ фіксує обов'язкові правила роботи з типами в проєкті.
Він описує, **як потрібно робити**, а не поточну реалізацію окремих модулів.

## Базові принципи

- Єдине джерело істини для API-контрактів: `Zod`-схеми.
- Типи не дублюються вручну, а виводяться через `z.infer`.
- Runtime-валидація і TypeScript-типи завжди мають походити з одного опису.
- Контролер відповідає за типізацію HTTP boundary.
- Сервіс відповідає за строгі сигнатури вхідних/вихідних даних.

## Обов'язковий workflow для нового endpoint

1. Описати вхід/вихід через `Zod`-схеми.
2. Вивести TS-типи через `z.infer`.
3. Для request body/query/params створити DTO через `createZodDto`.
4. На рівні контролера зафіксувати response-контракт через `@ZodResponse`.
5. У сервісі вказати явні сигнатури повернення (`Promise<ContractType>`).
6. Перевикористовувати спільні типи запиту й утиліти нормалізації замість локальних ad-hoc типів.

## Правила для request-типів

- Узагальнені request-типи зберігаються централізовано в спільному шарі типів.
- Локальні типи в контролерах дозволені лише як тимчасовий виняток.
- Якщо тип запиту потенційно потрібен у 2+ місцях, він одразу переноситься у спільний файл.
- Для авторизованих endpoint-ів використовуються окремі типи для `req.user`.
- Доступ до cookies/headers має бути типізованим, без `any` і без неявних `as` у бізнес-логіці.

## Правила для мережевих метаданих

- Значення на кшталт `ip`, `userAgent`, `forwarded-for` обробляються через спільну утиліту.
- Формат метаданих описується окремим інтерфейсом/типом.
- Контролери не дублюють парсинг headers вручну.
- Сервіс працює вже з нормалізованою типізованою структурою.

## Правила для сервісів

- Кожен публічний метод сервісу має явний тип повернення.
- Тип повернення сервісу має бути узгоджений з API-контрактом (через спільні схеми/типи).
- Заборонено повертати "вільні" об'єкти без контрактного типу.
- Внутрішні допоміжні структури також мають бути типізовані.

## Do / Don't

### Do

- Використовуй `z.infer` для побудови типів із схем.
- Винось повторно використовувані request-типи в спільний шар.
- Вказуй явні `Promise<...>` у сигнатурах сервісів.
- Нормалізуй технічні поля request через утиліти.

### Don't

- Не дублюй один і той самий контракт вручну в кількох місцях.
- Не залишай ad-hoc типи в контролері, якщо вони стають повторно використовуваними.
- Не змішуй бізнес-логіку з парсингом сирих HTTP headers.
- Не використовуй `any` для request/response даних.

## Чекліст перед merge

- Є `Zod`-схема для нового контракту.
- Типи виведені через `z.infer`.
- DTO створений через `createZodDto` (де потрібно).
- Контролер має `@ZodResponse` для endpoint-ів із відповіддю.
- Сервіс має явні сигнатури повернення.
- Нові request-типи/утиліти не дублюють наявні.
