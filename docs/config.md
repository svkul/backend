# Конфігурація

## Мета

Цей документ фіксує правила роботи з конфігурацією в проєкті.

## Базові принципи

- Конфігурація має бути централізованою і типізованою.
- Змінні оточення валідовуються на старті застосунку.
- Доступ до конфігу в коді відбувається через `ConfigService` і namespaced-конфіги.
- Пряме використання `process.env` у прикладному коді заборонене.
- Будь-яка нова змінна має бути описана в схемі валідації та у відповідному namespace.

## Обов'язковий workflow для нової змінної

1. Додати змінну в `.env.example` з коротким поясненням призначення.
2. Додати поле у схему валідації (`zod`) з коректним типом та обмеженнями.
3. За потреби додати `default(...)` тільки для безпечних і очікуваних значень.
4. Додати змінну у відповідний `registerAs(...)` namespace.
5. Оновити тип конфігурації через `ConfigType<typeof ...>` або суміжні типи.
6. Використовувати значення в модулях/сервісах тільки через `ConfigService`.
7. Якщо змінна критична, використовувати `getOrThrow(...)`.

## Правила структурування конфігів

- Групувати змінні за доменами (наприклад: `app`, `auth`, `oauth`, `db`, `web`).
- Не змішувати змінні різних доменів в одному namespace без потреби.
- Назви env-змінних мають бути однозначними та передбачуваними.
- Внутрішні назви полів у namespace мають бути читабельними (`camelCase`), без дублювання префіксів із env.

## Правила валідації

- Кожна env-змінна має валідуватися за типом (`string`, `number`, `boolean`, `url`, enum).
- Для числових значень задавати допустимі межі.
- Для URL використовувати явну перевірку формату.
- Для секретів заборонено задавати небезпечні дефолтні значення.
- Помилка валідації має зупиняти запуск застосунку.

## Правила використання в коді

- Читати конфіг на межі модуля/сервісу, а не в глибоких helper-функціях без потреби.
- Не дублювати ключі конфігів рядками по всьому коду.
- Перевага за типізованим доступом через namespace-ключі.
- Значення конфігу не повинні змінюватися під час runtime.

## Do / Don't

### Do

- Додавай змінну одночасно в `.env.example`, схему валідації та namespace.
- Використовуй строгі типи та перевірки формату.
- Використовуй `getOrThrow(...)` для обов'язкових значень.
- Оновлюй документацію при додаванні нової групи конфігурації.

### Don't

- Не використовуй `process.env` напряму в контролерах/сервісах.
- Не додавай змінні без валідації.
- Не зберігай секрети в коді або дефолтах.
- Не дублюй ту саму змінну в кількох namespace.

## Чекліст перед merge

- Нова змінна є в `.env.example`.
- Нова змінна є в `zod`-схемі валідації.
- Нова змінна підключена до правильного namespace.
- Код використовує `ConfigService`, без прямого `process.env`.
- Для критичних значень застосовано `getOrThrow(...)`.
- Документація оновлена, якщо змінено підхід або структуру.
